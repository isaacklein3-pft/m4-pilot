False-Positive Suppression Analysis

The false-positive prevention mechanism was subjected to 36 transient-blip scenarios — single-poll anomalies, sub-threshold CRS dips, and latency spikes shorter than the sustain window. Zero incorrect failovers were triggered.

The mechanism operates as follows: every trigger type requires the adverse condition to sustain across a minimum observation window before OCP initiates reassignment. CRS degradation requires one full 60-second scoring cycle in which the score remains below 40 — a single low poll does not qualify. Telemetry gaps require 90 consecutive seconds of silence — a late packet that arrives at second 88 resets the clock. Latency triggers require p95 above 400ms across a full 60-second window, not a single high-latency sample.

Critically, the OCP runs a three-poll confirmation buffer before broadcasting any weight update. A trigger must be confirmed across three consecutive 20-second OCP evaluation loops. This buffer eliminates routing churn from momentary network congestion or clock skew without introducing meaningful delay — the entire sustain window still resolves within the 30-second failover SLA because the confirm buffer runs concurrently with the sustain window, not after it.

Failover Stress Test Verdict OCP routing logic is production-ready. All 24 failover scenarios resolved within SLA. Zero false positives across 36 adversarial transient scenarios. Allocation safety constraints (60% cap, 2-operator minimum) held under all post-failover states. Stress test: PASSED.

Transition Framework — Live Institutional Validator Onboarding

The following framework governs the admission of the first live institutional validator. It is binding on all parties: network leadership, founding cohort operators, and the prospective institution. Deviation from any gate requires written authorization from the Network Governance Council.

Gate 1 — Operator Readiness Re-Verification

Before any institution-side onboarding begins, the target operator must pass a real-time readiness snapshot. This snapshot replicates the M4-1 audit criteria against the operator's live production environment, not the pilot environment.

  • ✓ Operator's DeploymentStatus is ACTIVE in middleware registry.
  • ✓ Telemetry stream producing valid CRS inputs for at least 72 consecutive hours.
  • ✓ CRS above 70 on 95% of evaluations across the 72-hour window.
  • ✓ p95 latency below 250ms (below the 400ms failover threshold, confirming operational headroom).
  • ✓ Operator signing ceremony endpoint live, accepting test envelope, rejecting replay attempt correctly.

Gate 2 — Institution Pre-Onboarding Verification

The institution completes a documented pre-flight before any signing ceremony is executed. Network leadership reviews and signs off before Gate 3 opens.

  • ✓ Custody environment confirmed: institution-owned HSM (FIPS 140-2 Level 3), air-gapped system, or institutional multisig with institution as policy owner. Cloud HSM rejected.
  • ✓ Maker, Checker, and Reviewer roles assigned to distinct staff with documented authorization matrix.
  • ✓ Institution nonce log initialized with 30-day retention policy confirmed.
  • ✓ Key generation performed inside custody boundary with no external participant.
  • ✓ Institution legal and compliance review complete; Letter of Agreement signed.
  • ✓ Emergency escalation contacts designated and tested (response time < 2 hours confirmed).

Gate 3 — Dry-Run Signing Ceremony

A non-live signing ceremony is executed with the live operator and the onboarding institution. This ceremony uses a production-format envelope but does not broadcast. All three MCR roles participate. Completion criteria must be met before the live registration ceremony proceeds.

  • ✓ Envelope delivery confirmed within 500ms to institution custody environment.
  • ✓ Institution Maker prepares payload; Checker verifies signature; Reviewer authorizes.
  • ✓ Signed blob returned to middleware and validated (Ed25519, canonical JSON, freshness window).
  • ✓ Replay attempt on dry-run envelope confirms ERR_NONCE_REPLAY issued correctly.
  • ✓ Institution Reviewer signs off on ceremony procedure documentation.

Gate 4 — Live Validator Registration

With Gates 1 through 3 satisfied, the live registration ceremony is authorized. Middleware constructs the validator registration payload using the confirmed production action class. Institution executes the signing ceremony. Middleware validates and broadcasts. OCP incorporates the new validator into the live routing graph.

  • ✓ Network Governance Council issues written onboarding authorization.
  • ✓ Live ceremony executed with all MCR roles present simultaneously.
  • ✓ Middleware broadcasts registration transaction; on-chain confirmation received.
  • ✓ OCP routing graph updated; new validator appears in telemetry within 60s.
  • ✓ Post-registration allocation weight confirmed below 60% cap.
  • ✓ 24-hour monitoring window initiated; no failover triggers expected.
Conditional Hold Provision If any gate criterion fails during a live onboarding attempt, the process halts and the operator returns to PENDING state. Re-entry to Gate 1 requires a 48-hour cooling period and a fresh 72-hour telemetry window. Exceptions require unanimous Network Governance Council approval.

Post-Onboarding Monitoring Requirements

The 24-hour post-registration monitoring window is mandatory for the first live institutional validator. During this window, OCP alert thresholds are tightened to 80% of normal trigger values (CRS floor raised to 48, latency SLA reduced to 320ms) to provide early warning before threshold breach. At the end of 24 hours, standard thresholds resume and the institution graduates to full production status.
